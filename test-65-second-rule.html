Why 65 Seconds Instead of 60?

  • Location sent every 60 seconds - Mobile app sends location updates
  • 5-second buffer - Accounts for network delays and processing time
  • Clock differences - Server and mobile device clocks may not be perfectly synchronized
  • Network latency - Internet connection delays can cause timing issues
  • Database processing - Time needed to process and store location data
  • Prevents false offline - Avoids marking representatives as offline due to timing issues
-- Database function with 65-second rule CREATE OR REPLACE FUNCTION get_representatives_with_online_status_60s() RETURNS TABLE (...) AS $$ BEGIN RETURN QUERY SELECT r.*, ll.timestamp AS last_seen, -- Online if location was sent within last 65 seconds (60s + 5s buffer) (ll.timestamp IS NOT NULL AND ll.timestamp > now() - INTERVAL '65 seconds') AS is_online FROM representatives r LEFT JOIN LATERAL ( SELECT * FROM representative_live_locations WHERE representative_id = r.id ORDER BY timestamp DESC LIMIT 1 ) ll ON true; END; $$ LANGUAGE plpgsql;
// JavaScript calculation with 65-second rule const isOnline = latestLocation ? ((new Date().getTime() - new Date(latestLocation.timestamp).getTime()) / (1000 * 60) <= 1.08) : false // 1.08 minutes = 65 seconds (60 + 5 buffer)
